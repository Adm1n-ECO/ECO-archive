## Architecture Rules (inherited from F4F — mandatory)

1. **CSS variables always** — never hardcode hex. All colors in `css/theme.css` as `var(--color-*)`.
2. **Supabase RLS mandatory** — every table has Row Level Security. Use `auth.jwt()->>'email'` not `auth.email()`.
3. **Edge functions JWT off** — all deployed with `--no-verify-jwt`. Auth handled manually in function body.
4. **No `.in()` with null** — use `.or('status.eq.pending,status.is.null')` pattern.
5. **Fetch before update** — read data you need BEFORE changing status fields (RLS blocks reads after).
6. **Apostrophes in JS** — never unescaped in single-quoted strings. Use array join pattern for long strings.
7. **`</script>` in JS strings** — always escape as `<\/script>`.
8. **DROP POLICY before CREATE** — PostgreSQL has no `CREATE POLICY IF NOT EXISTS`.
9. **`auth.jwt()->>'email'`** — never `auth.email()` in RLS policies.
10. **site-components.js** — every page loads it. Injects nav, footer, theme, referrer-policy meta.
11. **SQL migrations numbered** — sequential, `IF NOT EXISTS`, safe to re-run.
12. **Mobile menus hardcoded** — NOT generated by site-components.js.
13. **Canonical CSP** — copy exactly, never invent a variant (see Architecture section).

## REQ-004 · Authentication & Security

### 4.1 Auth method
- Primary: Email magic link (Supabase Auth)
- Secondary: Email + password
- Magic link redirect: always → `signin.html` (NOT `index.html`)
- `signin.html` handles `exchangeCodeForSession` and portal routing

### 4.2 RLS rules (mandatory on every table)
- Users can only read/write their own rows
- Pattern: `WHERE LOWER(email) = LOWER(auth.jwt()->>'email')`
- Never use `auth.email()` — returns NULL in JS client context
- Service role key (edge functions only) bypasses RLS
- `DROP POLICY IF EXISTS` then `CREATE POLICY` — no IF NOT EXISTS for policies

### 4.3 Privacy tiers per connection
- Each connection has a visibility level set by the owner
- Levels: `full` (name + contact + location) / `partial` (name + location) / `minimal` (name only) / `hidden` (on tree but anonymous)
- Privacy level stored on `connections.privacy_level` column (add to schema)

### 5.1 Stripe setup
- Test mode: `FMTO_TEST_MODE = true` in `js/auth.js` — flip to false before go-live
- Price IDs to be created in Stripe dashboard (FMTO-specific — never reuse F4F IDs)

### 5.2 Subscription flow
1. User signs up → lands on free (viewer) tier
2. Prompted to upgrade to Personal ($3.99/mo) to create household + build tree
3. Stripe Checkout → webhook → `users.subscription_tier` updated via edge function
4. Cancellation → downgrade to viewer, retain data, lose edit access

### 5.3 Stripe webhook edge function
- Function: `supabase/functions/stripe-webhook/index.ts`
- Events to handle: `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`
- Updates: `users.subscription_tier`, `users.subscription_status`, `users.stripe_customer_id`

## REQ-009 · Core Feature Specifications

### 9.1 Household node (the fundamental unit)
- A household = one or more people living together or grouped by family unit
- Can have a profile: name, photo, city/country, bio
- Members within a household each have their own user account
- Household is the node shown on the tree — not individual people

### 9.2 Tree view
- Interactive force-directed or hierarchical graph
- Nodes = households, edges = connections
- Each edge has a type (family / friend / school / work / custom)
- Clicking a node opens the household profile
- Zoom, pan, fit-to-screen controls

### 9.3 Segment toggle system (key differentiator)
- Segments are named, colored layers the user creates
- Any connection can belong to one or more segments
- User can toggle segments on/off to show/hide those connections
- Example segments: "Dad's side", "Mom's side", "School friends", "UK family"
- Segment state is per-user (your view, not shared)

### 9.4 Globe view (Phase 2 — key shareable feature)
- World map with household nodes plotted at lat/lng
- Lines connecting households in your network
- Visually shows the geographic spread of your network
- Screenshot-worthy — designed to be shared on social media
- Uses Mapbox or Leaflet.js

### 9.5 Network merge (consent-based)
- When User A invites User B who already has a tree:
  1. System detects overlap potential
  2. Presents merge proposal — specific connections to link
  3. Both parties must approve each merged connection
  4. No automatic overwrites — every merge is intentional

### 9.6 Life events (Phase 3)
- Events: birth, wedding, move, graduation, death, custom
- Owner marks event on their household
- Chooses which segments/connections to notify
- Notification goes via email (Resend) + in-app feed
- Not a social feed — deliberate broadcast, not continuous posting
